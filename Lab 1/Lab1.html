Bloc 1 — SQL vs NoSQL : Comprendre les différences

MICRO-COURS Durée : 20 min

1.1 Le monde des bases de données relationnelles (SQL)

Depuis les années 1970, la quasi-totalité des applications stockent leurs données dans des bases de données relationnelles (aussi appelées SQL databases). Le principe est simple : les données sont organisées en tables avec des lignes (chaque ligne représente un enregistrement) et des colonnes (chaque colonne représente un attribut). Par exemple, une table utilisateurs aurait les colonnes id, nom, email, date_creation. Chaque utilisateur occupe une ligne. Pour interroger ces tables, on utilise le langage SQL (Structured Query Language) : SELECT nom FROM utilisateurs WHERE id = 42;.

Les bases relationnelles imposent un schéma rigide : avant d'insérer des données, il faut définir précisément la structure de chaque table (noms de colonnes, types de données, contraintes). C'est comme un formulaire papier : si le formulaire n'a pas de case « numéro de téléphone », vous ne pouvez pas l'ajouter sans modifier le formulaire lui-même. Cette rigidité garantit la cohérence des données mais rend les modifications de structure coûteuses, surtout quand la base contient des millions de lignes.

Les bases SQL sont excellentes pour les données très structurées avec des relations entre tables (d'où le nom « relationnelle »). Par exemple, une table commandes est liée à une table clients via une clé étrangère. Cependant, elles peuvent devenir lentes lorsque le volume de données explose ou lorsque la structure des données varie beaucoup d'un enregistrement à l'autre — ce qui est précisément le cas des logs de sécurité.

1.2 L'émergence du NoSQL

Au début des années 2000, des entreprises comme Google, Amazon et Facebook ont constaté que les bases SQL classiques ne suffisaient plus pour gérer des milliards de requêtes par jour avec des structures de données très variées. Le terme NoSQL est apparu — il signifie « Not Only SQL », c'est-à-dire « pas seulement SQL ». Ce n'est pas un rejet du SQL, mais une alternative complémentaire.

Le NoSQL regroupe en réalité plusieurs familles de bases de données, chacune optimisée pour un type de problème particulier. Les quatre grandes familles sont : les bases Document (MongoDB, CouchDB) qui stockent des objets JSON flexibles ; les bases Clé-Valeur (Redis, DynamoDB) qui fonctionnent comme un dictionnaire géant ; les bases Colonnes larges (Cassandra, HBase) optimisées pour les écritures massives ; et les bases Graphe (Neo4j, ArangoDB) qui modélisent les relations complexes entre entités. Dans ce module, nous nous concentrons sur la famille Document avec MongoDB, car c'est la plus répandue et la plus pertinente pour les logs de cybersécurité.

La différence fondamentale est le schéma flexible : en NoSQL document, chaque enregistrement (appelé « document ») peut avoir une structure différente. Un log de connexion peut contenir un champ device, tandis qu'un log de pare-feu n'en a pas besoin. Pas besoin de modifier un schéma global. C'est comme passer d'un formulaire rigide à des fiches libres : chaque fiche peut contenir les informations pertinentes sans contrainte de format.

1.4 Exemple concret en cybersécurité

Imaginons un SIEM (Security Information and Event Management) qui collecte des logs de sources différentes : pare-feu, serveur web, antivirus, badge d'accès physique. En SQL, il faudrait soit créer une table par source (complexe à maintenir), soit créer une table unique avec énormément de colonnes dont la plupart seraient vides (gaspillage). En NoSQL Document, chaque log est un document autonome avec exactement les champs nécessaires. Un log pare-feu contiendra src_ip, dst_ip, port, protocol, tandis qu'un log badge contiendra badge_id, door, building. Les deux coexistent dans la même collection sans problème.

Figure 1 — Comparaison SQL (table rigide) vs NoSQL Document (documents flexibles)

Cette figure illustre la différence fondamentale entre les deux approches. À gauche, la table SQL impose une structure identique à toutes les lignes : même quand un champ n'est pas pertinent (par exemple, device pour un scan réseau), la colonne existe et contient NULL. C'est un gaspillage d'espace et une source de complexité dans les requêtes. À droite, chaque document JSON ne contient que les champs réellement utiles. Le document 3 ajoute un champ port qui n'existe pas dans les autres — c'est parfaitement valide. Cette flexibilité est particulièrement précieuse en cybersécurité, où les sources de logs ont des formats très différents les uns des autres.

À retenir

  • SQL = tables rigides, schéma défini à l'avance, excellent pour les données structurées et les transactions.
  • NoSQL Document = documents JSON flexibles, schéma libre, excellent pour les données hétérogènes et le volume.
  • NoSQL ne remplace pas SQL : ce sont des outils complémentaires. Le choix dépend du problème à résoudre.
  • En cybersécurité, les logs hétérogènes sont un cas d'usage idéal pour le NoSQL document.

Erreur fréquente : « NoSQL est plus moderne donc meilleur que SQL »

Pourquoi c'est dangereux : Choisir NoSQL pour une application bancaire qui nécessite des transactions ACID strictes peut mener à des incohérences de données catastrophiques (double débit, perte de transaction). En sécurité, utiliser le mauvais outil peut aussi signifier perdre des logs critiques lors d'un incident. Le bon réflexe : analyser le besoin avant de choisir la technologie.

Bloc 2 — JSON & le modèle Document

MICRO-COURS Durée : 15 min

2.1 Qu'est-ce que JSON ?

JSON (JavaScript Object Notation) est un format de texte léger pour représenter des données structurées. Il est lisible par les humains et facile à traiter par les machines. Un objet JSON est délimité par des accolades { } et contient des paires clé : valeur. Les clés sont toujours des chaînes de caractères entre guillemets. Les valeurs peuvent être des chaînes ("alice"), des nombres (22), des booléens (true/false), des tableaux ([1, 2, 3]), d'autres objets JSON (imbrication), ou null.

Pensez au JSON comme à une fiche d'incident que remplirait un analyste SOC (Security Operations Center). Chaque fiche a un format libre : l'analyste note les informations pertinentes sans être contraint par un formulaire rigide. Une fiche pour une tentative de brute-force contiendra l'IP source, le nombre de tentatives et les comptes ciblés, tandis qu'une fiche pour un malware contiendra le hash du fichier, le chemin et le moteur antivirus qui l'a détecté.

Voici un exemple de document JSON représentant un log de sécurité :

{
  "timestamp": "2026-02-25T09:15:30Z",
  "src_ip": "192.168.1.105",
  "user": "jdupont",
  "action": "login_failed",
  "status": "failure",
  "device": "workstation-PC42",
  "geo": {
    "country": "FR",
    "city": "Paris"
  },
  "severity": "warning"
}

2.2 Le modèle Document dans MongoDB

MongoDB organise les données en trois niveaux hiérarchiques. Au sommet, une base de données (database) regroupe des données liées à un même projet ou domaine — par exemple noseclab pour notre laboratoire de sécurité. À l'intérieur d'une base, les données sont organisées en collections — l'équivalent approximatif d'une table SQL, mais sans schéma imposé. Par exemple, la collection security_logs. Enfin, chaque enregistrement individuel est un document — un objet JSON (techniquement BSON, une version binaire de JSON utilisée par MongoDB pour l'efficacité).

Chaque document possède obligatoirement un champ _id qui sert d'identifiant unique. Si vous n'en fournissez pas lors de l'insertion, MongoDB en génère un automatiquement sous la forme d'un ObjectId, une chaîne de 24 caractères hexadécimaux qui encode la date de création, un identifiant machine, un identifiant de processus et un compteur. Cet _id est immuable : une fois créé, il ne peut pas être modifié.

Figure 3 — Hiérarchie MongoDB : Base de données → Collections → Documents

Cette figure montre les trois niveaux de MongoDB. La base noseclab est le conteneur principal. Elle contient plusieurs collections thématiques. La collection security_logs contient nos documents de logs — chacun avec sa propre structure. Notez que les deux documents illustrés n'ont pas exactement les mêmes champs : le document 2 a un champ port que le document 1 n'a pas, et inversement pour le champ user. C'est la puissance du schéma flexible. Chaque document est identifié de manière unique par son _id (ObjectId).

À retenir

  • Base de données → conteneur logique (ex : noseclab).
  • Collection → groupe de documents similaires (ex : security_logs), équivalent souple d'une table SQL.
  • Document → objet JSON individuel = un enregistrement (ex : un log).
  • _id → identifiant unique auto-généré (ObjectId) ou défini manuellement. Immuable.
  • BSON → format binaire de JSON utilisé en interne par MongoDB (plus rapide, supporte plus de types).

Erreur fréquente : « Pas de schéma = pas de règles = je mets n'importe quoi »

Pourquoi c'est dangereux : L'absence de schéma imposé ne signifie pas qu'il ne faut pas de convention. Si chaque développeur utilise un nom de champ différent (srcIp, src_ip, source_ip, ip_source), les requêtes de détection d'incident deviennent un cauchemar. En sécurité, un log mal nommé peut passer entre les mailles d'une règle de détection et laisser une attaque non détectée. Bonne pratique : définir une convention de nommage documentée, même si MongoDB ne l'impose pas techniquement.

Bloc 3 — Architecture du Lab (Docker + MongoDB)

MICRO-COURS Durée : 15 min

3.1 Pourquoi Docker ?

Docker est un outil qui permet de lancer des applications dans des conteneurs — des environnements isolés et reproductibles. Imaginez un conteneur comme une « boîte » qui contient tout ce dont une application a besoin pour fonctionner : le logiciel, ses dépendances, sa configuration. L'avantage majeur : que vous soyez sur Windows, Linux ou macOS, le conteneur se comporte de manière identique. Plus de « ça marche sur ma machine mais pas sur la tienne ».

Pour notre lab, nous allons lancer MongoDB dans un conteneur Docker. Cela nous évite d'installer MongoDB directement sur votre machine (ce qui peut être complexe et varie selon le système d'exploitation). En une seule commande, vous aurez un serveur MongoDB opérationnel. Et quand le TP est terminé, vous pouvez supprimer le conteneur sans laisser de traces sur votre système.

L'architecture est simple : votre terminal (ligne de commande) communique avec le client mongosh (MongoDB Shell), qui lui-même se connecte au serveur MongoDB tournant dans un conteneur Docker. Le serveur écoute par défaut sur le port 27017. C'est exactement comme une architecture client-serveur classique, sauf que le serveur est dans un conteneur.

🔧 Vérification Docker — À lire AVANT le TP

Ouvrez un terminal et tapez :

docker --version

Résultat attendu : Docker version 24.x.x (ou supérieur).

Si Docker n'est pas installé :

  • Windows : Installez Docker Desktop (nécessite WSL2 activé).
  • macOS : Installez Docker Desktop via le .dmg officiel.
  • Linux : sudo apt install docker.io (Debian/Ubuntu) ou sudo dnf install docker (Fedora).
  • Alternative sans Docker : Utilisez MongoDB Atlas (version cloud gratuite) — l'enseignant fournira les instructions si nécessaire.

Vérifiez aussi que Docker tourne : docker ps (doit afficher un tableau, même vide).

Figure 2 — Architecture du TP : PC → Docker → MongoDB → mongosh

Votre terminal envoie des commandes à Docker, qui fait tourner un conteneur contenant le serveur MongoDB (mongod). Le serveur écoute sur le port 27017 à l'intérieur du conteneur. Grâce au port mapping (-p 27017:27017), ce port est accessible depuis votre machine hôte. Le client mongosh se connecte à localhost:27017 pour interagir avec la base. Les données sont stockées dans un volume Docker (/data/db), ce qui permet de les conserver même si le conteneur est redémarré. En cybersécurité, notez que ce port mapping est une surface d'attaque : si votre machine est sur un réseau, le port 27017 peut être accessible de l'extérieur.

À retenir

  • Docker = conteneur isolé, reproductible, multiplateforme.
  • mongod = processus serveur MongoDB (celui qui stocke et gère les données).
  • mongosh = client shell interactif pour envoyer des commandes à MongoDB.
  • Port 27017 = port par défaut de MongoDB. À protéger absolument en production.
  • Le port mapping Docker rend un service interne accessible depuis l'extérieur → impact sécurité direct.

Erreur fréquente : « Le conteneur tourne = la base est sécurisée »

Pourquoi c'est dangereux : Docker isole les processus mais n'ajoute aucune sécurité applicative. Par défaut, MongoDB dans Docker démarre sans authentification. Quiconque peut se connecter au port 27017 a un accès total aux données — lecture, écriture, suppression. C'est exactement ce qui s'est passé en 2017 lors des MongoDB Ransomware attacks, où des milliers de bases exposées sans auth ont été vidées et rançonnées.

Bloc 4 — Connexion mongosh, base, collection & dataset security_logs

MICRO-COURS Durée : 15 min

4.1 Naviguer dans MongoDB avec mongosh

Une fois connecté à mongosh, vous interagissez avec MongoDB via des commandes JavaScript. Voici les commandes de navigation essentielles : show dbs affiche toutes les bases de données existantes, use nomDeLaBase bascule vers une base (ou la crée si elle n'existe pas encore — mais elle ne sera réellement créée que lorsqu'on y insérera des données), et show collections liste les collections de la base courante.

MongoDB suit une philosophie « lazy creation » (création paresseuse) : les bases et les collections ne sont physiquement créées qu'au moment de la première insertion de données. Cela signifie que use noseclab ne crée pas immédiatement la base — elle « existera » seulement après votre premier insertOne() ou insertMany(). C'est un comportement important à comprendre car il peut être surprenant pour les habitués du SQL (où il faut explicitement faire CREATE DATABASE puis CREATE TABLE).

Le dataset fil conducteur de ce module est une collection security_logs. Chaque document représente un événement de sécurité avec les champs suivants : timestamp (horodatage ISO 8601), src_ip (adresse IP source), user (identifiant utilisateur, peut être absent), action (type d'événement : login, logout, scan, etc.), status (success, failure, detected), device (appareil utilisé), geo (objet imbriqué : country, city), severity (info, warning, high, critical). Ce jeu de données simule un mini-SIEM que vous pourriez rencontrer en stage ou en entreprise.

À retenir

  • show dbs → lister les bases. use <nom> → basculer vers une base.

Bloc 5 — CRUD : Create, Read, Update, Delete

MICRO-COURS Durée : 15 min

5.1 Les quatre opérations fondamentales

Toute base de données, qu'elle soit SQL ou NoSQL, repose sur quatre opérations fondamentales regroupées sous l'acronyme CRUD : Create (créer/insérer), Read (lire/interroger), Update (mettre à jour/modifier), Delete (supprimer). Dans MongoDB, chacune de ces opérations a une méthode dédiée qui s'appelle sur l'objet collection via db.nomCollection.methode().

Create — Pour insérer des données, on utilise insertOne() pour un seul document ou insertMany() pour plusieurs documents à la fois (comme on l'a fait au TP 4). Chaque document inséré reçoit automatiquement un _id unique si vous n'en fournissez pas. En cybersécurité, l'insertion correspond à l'arrivée d'un nouveau log dans le système : chaque événement détecté est « inséré » dans la base.

Read — Pour lire des données, on utilise find() (plusieurs documents) ou findOne() (un seul). On peut passer un filtre en paramètre : find({ status: "failure" }) retourne tous les documents où le statut est « failure ». Sans filtre, find() retourne tous les documents. C'est l'équivalent de SELECT * FROM table WHERE ... en SQL. En sécurité, c'est l'opération la plus fréquente : rechercher des événements suspects, filtrer les tentatives échouées, tracer l'activité d'un utilisateur.

Update — Pour modifier des documents existants, on utilise updateOne() (le premier qui correspond au filtre) ou updateMany() (tous ceux qui correspondent). La syntaxe utilise des opérateurs comme $set (modifier un champ), $unset (supprimer un champ), $inc (incrémenter). Exemple : updateOne({ user: "bob" }, { $set: { severity: "critical" } }). En sécurité, on met à jour la sévérité d'un incident, on ajoute des notes d'investigation, on change le statut d'une alerte.

Delete — Pour supprimer, on utilise deleteOne() ou deleteMany(). Attention : la suppression est irréversible ! deleteMany({}) sans filtre supprime tous les documents de la collection. En cybersécurité, la suppression de logs est un sujet sensible : un attaquant qui obtient un accès à la base pourrait effacer les traces de son intrusion. C'est pourquoi la protection en écriture/suppression est cruciale.

Figure 4 — Les quatre opérations CRUD vers MongoDB

Le client mongosh envoie des opérations CRUD au serveur MongoDB. Chaque opération agit sur une collection spécifique. Create ajoute de nouveaux documents, Read les interroge (avec ou sans filtre), Update modifie des documents existants en utilisant des opérateurs ($set, $inc...), et Delete les supprime. En cybersécurité, l'accès en écriture (Update, Delete) est le plus sensible : un attaquant pourrait modifier des logs pour masquer une intrusion ou supprimer les preuves d'une activité malveillante. C'est pourquoi le contrôle d'accès granulaire est essentiel.

À retenir

  • Create : insertOne(doc) / insertMany([doc1, doc2, ...])
  • Read : find(filtre) / findOne(filtre)
  • Update : updateOne(filtre, { $set: { champ: valeur } }) / updateMany(...)
  • Delete : deleteOne(filtre) / deleteMany(filtre)
  • deleteMany({}) = supprime TOUT. Ne jamais exécuter sans réfléchir !

Erreur fréquente : Oublier $set dans un update

Écrire updateOne({ user: "alice" }, { severity: "critical" }) sans $set ne met pas à jour le champ — cela remplace tout le document par { severity: "critical" } ! Tous les autres champs sont perdus.

Pourquoi c'est dangereux : En sécurité, un remplacement accidentel de document signifie la perte de preuves (logs d'incident). Toujours utiliser $set pour modifier un champ spécifique.

Bloc 6 — Requêtes utiles : filtres, tri, limite, projection

MICRO-COURS Durée : 10 min

6.1 Opérateurs de comparaison

MongoDB fournit des opérateurs puissants pour construire des filtres précis. Les plus courants sont : $eq (égal, implicite), $ne (différent de), $gt (supérieur à), $gte (supérieur ou égal), $lt (inférieur à), $lte (inférieur ou égal), $in (dans une liste de valeurs), $nin (pas dans la liste). On les utilise dans le filtre : find({ severity: { $in: ["high", "critical"] } }).

6.2 Opérateurs logiques

Pour combiner des conditions : $and (toutes les conditions vraies — implicite quand on met plusieurs champs), $or (au moins une condition vraie), $not (négation). Exemple : find({ $or: [{ severity: "critical" }, { action: "port_scan" }] }).

6.3 Tri, limite et projection

Tri (sort) : .sort({ timestamp: -1 }) trie par ordre décroissant (les plus récents d'abord). 1 = croissant, -1 = décroissant.

Limite (limit) : .limit(5) ne retourne que les 5 premiers résultats. Très utile pour voir les derniers événements.

Projection : Le second argument de find() permet de choisir quels champs afficher. find({}, { src_ip: 1, action: 1, _id: 0 }) n'affiche que src_ip et action, en masquant le _id. 1 = inclure, 0 = exclure. On ne peut pas mélanger inclusion et exclusion (sauf pour _id).

Chaînage : On peut chaîner ces méthodes : db.security_logs.find({ status: "failure" }).sort({ timestamp: -1 }).limit(3) → les 3 derniers échecs, les plus récents en premier.

À retenir

  • Opérateurs de comparaison : $gt, $lt, $gte, $lte, $in, $ne.
  • Opérateurs logiques : $and (implicite), $or, $not.
  • .sort({ champ: -1 }) → tri décroissant. .limit(n) → limiter les résultats.
  • Projection : 2ème argument de find(). 1 = inclure, 0 = exclure.

Erreur fréquente : Oublier la projection et exposer des données sensibles

Pourquoi c'est dangereux : Dans une API qui interroge MongoDB, si le développeur fait find({}) sans projection, tous les champs sont renvoyés au client — y compris des données sensibles (IP internes, noms d'utilisateurs, tokens). Toujours utiliser la projection pour limiter les champs exposés dans une réponse API. C'est le principe du moindre privilège appliqué aux données.

Bloc 7 — Sécurité « by default » : surface d'attaque MongoDB

MICRO-COURS SÉCURITÉ Durée : 15 min

7.1 Le problème des configurations par défaut

Lorsqu'on installe un logiciel serveur — que ce soit une base de données, un serveur web ou un service réseau — la configuration par défaut est généralement conçue pour faciliter la prise en main, pas pour être sécurisée. Les éditeurs veulent que le produit « marche tout de suite » sans friction. C'est compréhensible d'un point de vue ergonomique, mais c'est une catastrophe du point de vue sécurité. MongoDB ne fait pas exception à cette règle.

Historiquement (avant la version 3.6), MongoDB écoutait par défaut sur toutes les interfaces réseau (0.0.0.0), ce qui signifie que n'importe quelle machine sur le réseau — ou pire, sur Internet — pouvait tenter de se connecter. De plus, l'authentification était désactivée par défaut : aucun mot de passe n'était requis pour lire, écrire ou supprimer des données. Résultat : en 2017, plus de 28 000 bases MongoDB exposées sur Internet ont été victimes d'attaques de type ransomware. Les attaquants se connectaient, supprimaient toutes les données, et laissaient un message demandant une rançon en Bitcoin.

Depuis la version 3.6, MongoDB a amélioré les choses : par défaut, le serveur écoute uniquement sur localhost (127.0.0.1), ce qui empêche les connexions depuis d'autres machines. Cependant, quand on utilise Docker avec un port mapping (-p 27017:27017), on contourne cette protection : le port devient accessible depuis l'extérieur du conteneur, potentiellement depuis tout le réseau local, voire Internet si la machine est directement exposée. De plus, l'image Docker officielle de MongoDB démarre sans authentification activée pour simplifier le développement. C'est exactement notre configuration de TP !

En tant que futurs professionnels de la cybersécurité, vous devez développer un réflexe : à chaque fois que vous déployez un service, posez-vous trois questions fondamentales. Qui peut se connecter ? (surface réseau : interfaces, ports, pare-feu). Qui est authentifié ? (identité : utilisateurs, mots de passe, certificats). Qui est autorisé à faire quoi ? (permissions : lecture seule, écriture, administration). Ce triptyque réseau → authentification → autorisation s'applique à MongoDB comme à n'importe quel autre service.

7.2 Les trois vecteurs d'exposition

VecteurConfiguration dangereuseConfiguration sécuriséeRisque concret

Bind address bindIp: 0.0.0.0 (toutes les interfaces) bindIp: 127.0.0.1 (localhost uniquement) Le serveur accepte les connexions depuis n'importe quelle IP

C'est un choix pédagogique : cela simplifie la prise en main et nous permet de nous concentrer sur l'apprentissage de MongoDB. Cependant, vous devez garder à l'esprit que cette configuration est strictement interdite en production. Le bloc suivant (Mini-TP 7) vous guidera pour vérifier concrètement cette exposition — sans attaquer, simplement en constatant les faits.

🛡️ Règles d'or — Sécurité NoSQL (niveau débutant)

  1. Toujours activer l'authentification (--auth) : aucun accès anonyme ne doit être possible.
  2. Limiter l'interface d'écoute : bindIp: 127.0.0.1 ou uniquement les interfaces nécessaires.
  3. Restreindre le port mapping Docker : utiliser -p 127.0.0.1:27017:27017 au lieu de -p 27017:27017.
  4. Créer des utilisateurs avec le moindre privilège : un utilisateur « lecture seule » pour les analystes, un utilisateur « admin » uniquement pour l'administration.
  5. Ne jamais exposer MongoDB directement sur Internet : utiliser un VPN, un tunnel SSH ou un réseau privé.
  6. Chiffrer les communications : activer TLS/SSL pour les connexions client-serveur.
  7. Surveiller et auditer : activer les logs d'audit MongoDB pour tracer qui fait quoi.

À retenir

  • La configuration par défaut de MongoDB (surtout via Docker) est non sécurisée : pas d'auth, port potentiellement exposé.
  • En 2017, 28 000+ bases MongoDB ont été rançonnées à cause de mauvaises configurations.
  • Le triptyque de sécurité : Réseau (qui peut se connecter ?) → Authentification (qui est-il ?) → Autorisation (que peut-il faire ?).
  • Docker -p contourne l'isolation réseau : le port mapping rend le service accessible.

Erreur fréquente : « C'est juste un lab / un serveur de dev, pas besoin de sécuriser »

Pourquoi c'est dangereux : Les serveurs « temporaires » ou « de développement » sont souvent oubliés et laissés en ligne pendant des mois. Ils deviennent des portes d'entrée idéales pour un attaquant : pas de surveillance, pas de patches, pas d'auth. C'est exactement comme laisser une porte de service non verrouillée dans un bâtiment « parce qu'elle ne sert plus ». Un scan réseau (Shodan, Censys) peut les trouver en quelques secondes. Sécurisez tout, même les labs.

MINI-TP 7 SÉCURITÉ Vérification de la surface d'attaque

Objectif : Constater (sans attaquer) que notre MongoDB est exposé, comprendre les risques.

Durée estimée : 15 minutes

Prérequis : Conteneur mongo-noseclab en fonctionnement.

Étape 1 — Vérifier le port exposé

Depuis votre terminal (en dehors de mongosh), vérifiez que le port 27017 est ouvert :

Option A — Avec nmap (si installé) :

nmap -p 27017 localhost
Résultat attendu : 
PORT      STATE SERVICE
27017/tcp open  mongod

Le port est ouvert ! Toute machine pouvant atteindre votre IP peut potentiellement s'y connecter.

Option B — Sans nmap, avec curl :

curl -s http://localhost:27017
Résultat attendu : It looks like you are trying to access MongoDB over HTTP on the native driver port.

Ce message confirme qu'un service MongoDB répond sur ce port. Un attaquant obtient cette même information.

Option C — Avec PowerShell (Windows) :

Test-NetConnection -ComputerName localhost -Port 27017
Résultat attendu : TcpTestSucceeded : True

Étape 2 — Vérifier l'absence d'authentification

Connectez-vous à mongosh sans fournir aucun identifiant :

docker exec -it mongo-noseclab mongosh --eval "db.adminCommand({ listDatabases: 1 })"
Résultat attendu : 
{
  databases: [
    { name: 'admin', sizeOnDisk: ..., empty: false },
    { name: 'config', sizeOnDisk: ..., empty: false },
    { name: 'local', sizeOnDisk: ..., empty: false },
    { name: 'noseclab', sizeOnDisk: ..., empty: false }
  ],
  totalSize: ...,
  ok: 1
}

Vous avez listé toutes les bases sans aucun mot de passe. Un attaquant pourrait faire exactement la même chose. Il verrait votre base noseclab, s'y connecterait, et pourrait lire, modifier ou supprimer tous vos logs de sécurité.

Étape 3 — Vérifier le binding réseau du conteneur

docker port mongo-noseclab
Résultat attendu : 27017/tcp -> 0.0.0.0:27017

0.0.0.0 signifie « toutes les interfaces réseau ». Le port est accessible depuis l'extérieur de la machine (réseau local). Si cette machine avait une IP publique, MongoDB serait accessible depuis Internet !

Étape 4 — Comparer avec une configuration sécurisée

Voici comment on devrait lancer MongoDB en production (ne l'exécutez pas, c'est pour comprendre la différence) :

# Configuration sécurisée (NE PAS exécuter maintenant)
docker run -d \
  --name mongo-secure \
  -p 127.0.0.1:27017:27017 \
  -e MONGO_INITDB_ROOT_USERNAME=admin \
  -e MONGO_INITDB_ROOT_PASSWORD=MotDePasseF0rt!2026 \
  mongo:7 --auth

Différences clés :

  • 127.0.0.1:27017:27017 → port accessible uniquement depuis localhost.
  • MONGO_INITDB_ROOT_USERNAME/PASSWORD → crée un utilisateur admin au démarrage.
  • --auth → active l'authentification obligatoire.

D. Glossaire NoSQL

TermeDéfinition simple
NoSQL« Not Only SQL » — famille de bases de données non relationnelles, optimisées pour des cas d'usage spécifiques (volume, flexibilité, performance).
DocumentUn enregistrement individuel dans MongoDB, stocké au format JSON/BSON. Équivalent d'une ligne en SQL, mais avec un schéma flexible.
CollectionUn groupe de documents dans MongoDB. Équivalent souple d'une table en SQL (pas de schéma imposé).
Base de données (Database)Conteneur logique regroupant plusieurs collections liées à un même projet ou domaine.
BSONBinary JSON — format binaire utilisé en interne par MongoDB pour stocker les documents. Plus rapide que JSON texte et supporte des types supplémentaires (Date, ObjectId, etc.).
ObjectIdIdentifiant unique de 12 octets (24 caractères hexadécimaux) généré automatiquement par MongoDB pour chaque document. Encode un timestamp, un identifiant machine, un PID et un compteur.
_idChamp obligatoire de chaque document MongoDB, servant de clé primaire. Immuable une fois créé.
CRUDCreate, Read, Update, Delete — les quatre opérations fondamentales sur les données.
mongodLe processus serveur (daemon) de MongoDB. C'est lui qui gère le stockage, les requêtes et les connexions.
mongoshMongoDB Shell — client interactif en ligne de commande pour envoyer des commandes à MongoDB.
Schéma flexibleCaractéristique des bases NoSQL document : chaque document d'une même collection peut avoir des champs différents, sans déclaration préalable.
ProjectionTechnique permettant de sélectionner quels champs d'un document sont retournés par une requête (1 = inclure, 0 = exclure).
OpérateurMot-clé préfixé par $ utilisé dans les requêtes MongoDB pour définir des conditions ($gt, $set, $in, etc.).
Port mappingFonctionnalité Docker (-p) qui rend un port interne au conteneur accessible depuis la machine h��te.
Bind addressL'adresse IP sur laquelle un serveur écoute les connexions. 127.0.0.1 = localhost uniquement. 0.0.0.0 = toutes les interfaces (dangereux).
SIEMSecurity Information and Event Management — système centralisant la collecte, l'analyse et la corrélation des logs de sécurité.
SOCSecurity Operations Center — centre opérationnel de surveillance et de réponse aux incidents de sécurité.
Surface d'attaqueEnsemble des points d'entrée qu'un attaquant pourrait exploiter pour compromettre un système (ports ouverts, services exposés, etc.).
Brute-forceAttaque consistant à tester un grand nombre de mots de passe ou combinaisons pour deviner des identifiants.
ShardingTechnique de scalabilité horizontale consistant à répartir les données sur plusieurs serveurs (sera vu en séances suivantes).
