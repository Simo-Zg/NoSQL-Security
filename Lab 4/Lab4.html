RRappel Séances 1–3 et motivation

Les trois premières séances ont posé les bases opérationnelles : installation de MongoDB sous Docker, opérations CRUD avec mongosh, modélisation par embedding et référence, requêtes d'agrégation, indexation — tout cela sur la base soc_lite. La Séance 3 a introduit le hardening infrastructure : authentification SCRAM, RBAC (rôles, utilisateurs limités), restriction réseau (bindIp), et gestion des secrets.

⚠ Pourquoi passer au niveau application ?

Le hardening infrastructure protège MongoDB contre une connexion directe non autorisée. Mais dans la réalité, MongoDB n'est jamais exposé directement aux utilisateurs finaux : une API (Node.js, Python, Java…) fait l'intermédiaire. C'est elle qui reçoit les saisies utilisateur, construit les filtres, et interroge la base.

Si cette API passe les données utilisateur directement dans un find() sans validation, le hardening infrastructure devient insuffisant : l'attaquant manipule l'API légitime sans jamais toucher MongoDB directement. L'attaquant n'a pas besoin de connaître les identifiants MongoDB — il lui suffit d'envoyer une requête HTTP malformée à l'API. C'est ce problème que cette séance traite.

💡 Analogie — La porte blindée et la fenêtre ouverte

Le hardening de Séance 3 revient à poser une porte blindée sur votre serveur MongoDB. Mais si l'API Node.js est une fenêtre ouverte que tout le monde peut atteindre depuis internet, l'attaquant n'a pas besoin de forcer la porte : il passe par la fenêtre et manipule le majordome (l'API) pour lui faire ouvrir le coffre (MongoDB). La sécurité applicative, c'est sécuriser la fenêtre.

1Comprendre NoSQL Injection

1.1 Définition et comparaison avec l'injection SQL

📖 Définition — NoSQL Injection

Une injection NoSQL est une attaque où un attaquant insère des opérateurs ou des structures JSON dans une entrée applicative afin de modifier la logique d'une requête MongoDB.

Contrairement à l'injection SQL (qui casse une chaîne de caractères pour y glisser du code SQL brut), l'injection NoSQL exploite le fait que les filtres MongoDB sont déjà des objets JSON. Si l'application place directement l'objet envoyé par l'utilisateur dans un find(), l'attaquant peut y inclure des opérateurs MongoDB ($ne, $regex, $gt…) qui en changent radicalement le sens — sans casser de syntaxe.

Comparaison SQL Injection vs NoSQL Injection

AspectSQL InjectionNoSQL Injection (MongoDB)
Vecteur d'attaque Chaîne de caractères concaténée dans une requête SQL Objet JSON inséré dans un filtre find()
Payload typique ' OR '1'='1 {"$ne":""} ou {"$regex":".*"}
Mécanisme Cassure de la syntaxe SQL pour injecter du code SQL Injection d'opérateurs dans l'objet filtre JSON, sans cassure de syntaxe
Requêtes préparées Protection efficace (les paramètres sont échappés) Concept non applicable directement — la défense est la validation de type + safeFilter
Défense principale Requêtes préparées / paramétrisées Validation de type + construction explicite du filtre
⚠ Erreur fréquente des développeurs

Beaucoup de développeurs pensent que MongoDB les protège naturellement des injections car "ce n'est pas du SQL". C'est une idée reçue dangereuse. La vulnérabilité n'est pas dans la base de données — elle est dans le code applicatif qui construit les filtres sans les valider. MongoDB exécute fidèlement le filtre qu'il reçoit, légitime ou injecté : il n'y a aucune distinction côté base.

1.2 Le mécanisme technique : comment l'injection se produit

Voici l'endpoint vulnérable typique pour rechercher des logs dans l'API soc_lite :

// POST /logs/search — VERSION VULNÉRABLE (server.js)
app.post('/logs/search', async (req, res) => {
  // ❌ req.body passé DIRECTEMENT dans find() — JAMAIS faire ça
  const results = await db.collection('security_logs')
                          .find(req.body)   // ❌ l'objet utilisateur devient le filtre
                          .limit(20)
                          .toArray();
  res.json({ count: results.length, data: results });
});

Cas normal : l'utilisateur envoie {"src_ip":"192.168.1.10"} → MongoDB filtre les logs de cette IP. ✔

Cas injecté : l'attaquant envoie {"src_ip":{"$regex":""}} → la regex vide correspond à toute chaîne → tous les logs sont retournés. ✘

🔍 Ce qui se passe dans MongoDB

MongoDB reçoit le filtre { src_ip: { $regex: "" } }. Il l'exécute littéralement : "retourne tous les documents où src_ip correspond à l'expression régulière vide". Une regex vide matche toutes les chaînes — donc tous les documents. MongoDB ne peut pas distinguer un filtre légitime d'un filtre injecté. La responsabilité de la validation appartient entièrement au code applicatif.

Le même problème affecte l'endpoint de login :

// POST /auth/login — VERSION VULNÉRABLE
app.post('/auth/login', async (req, res) => {
  const { username, password } = req.body;
  // ❌ Si password est {"$ne":""} au lieu d'une chaîne,
  //    MongoDB cherche "tous les utilisateurs dont le mot de passe ≠ ''"
  //    → retourne n'importe quel compte sans connaître son mot de passe !
  const user = await db.collection('users')
                       .findOne({ username, password });  // ❌
  if (user) res.json({ success: true, message: `Bienvenue ${user.username}`, role: user.role });
  else res.status(401).json({ success: false, message: 'Identifiants incorrects' });
});

🔲 Checkpoint 1 — Vérification de compréhension

Q1. Dans l'endpoint POST /logs/search, quel est le problème fondamental de sécurité ?

  • A) MongoDB est mal configuré pour rejeter les opérateurs dans les filtres.
  • B) L'objet req.body est passé directement à find() sans validation ni filtrage.
  • C) Express ne supporte pas les requêtes POST avec un corps JSON.
  • D) La collection security_logs n'est pas indexée correctement.
✅ B est correct. Le problème est entièrement dans le code Node.js : req.body est utilisé comme filtre sans aucune vérification. MongoDB exécute fidèlement le filtre qu'il reçoit, qu'il soit légitime ou injecté — il ne peut pas faire la distinction.

Q2. En quoi l'injection NoSQL diffère-t-elle fondamentalement de l'injection SQL ?

  • A) L'injection NoSQL est généralement moins dangereuse car MongoDB n'a pas de transactions.
  • B) L'injection SQL utilise des objets JSON ; l'injection NoSQL, des chaînes de caractères.
  • C) L'injection NoSQL exploite des opérateurs JSON dans l'objet filtre, sans cassure de syntaxe.
  • D) Les deux attaques sont identiques, seul le moteur de base de données change.
✅ C est correct. En SQL, on casse une chaîne pour y insérer du code SQL brut. Dans MongoDB, les filtres sont déjà des objets JSON : l'attaquant y insère des opérateurs comme $ne ou $regex qui modifient la logique sans aucune cassure de syntaxe. C'est pourquoi les techniques de protection SQL ne s'appliquent pas directement.

2Démonstration contrôlée des injections NoSQL

🔒 Cadre éthique — À lire avant de continuer

Toutes les démonstrations s'effectuent exclusivement sur votre machine locale, sur une base de données créée pour ce cours. Il est rigoureusement interdit de reproduire ces techniques sur tout système externe, même avec autorisation présumée. L'objectif pédagogique est de comprendre le mécanisme pour mieux le défendre — pas de l'exploiter.

2.1 Injection via $ne — Contournement d'authentification

📖 L'opérateur MongoDB $ne (Not Equal)

{"champ":{"$ne":valeur}} sélectionne tous les documents où champ est différent de valeur.

Usage légitime : db.logs.find({ status: { $ne: "closed" } }) → retourne tous les logs non fermés.

Injecté à la place d'un mot de passe : {"password":{"$ne":""}} → sélectionne tous les utilisateurs dont le mot de passe est différent d'une chaîne vide, c'est-à-dire pratiquement tous les utilisateurs ayant un mot de passe.

Sur l'endpoint POST /auth/login, si l'attaquant envoie {"username":"alice","password":{"$ne":""}}, le filtre MongoDB devient :

// Ce que l'API vulnérable construit involontairement :
{ username: "alice", password: { $ne: "" } }

// Ce que MongoDB comprend :
// "retourne le document dont username = 'alice'
//  ET dont password est DIFFÉRENT de la chaîne vide"
// = "alice" dont le mot de passe n'est pas vide
// = Alice tout court (puisqu'elle a un mot de passe)
// → Login réussi SANS connaître le mot de passe !
💡 Impact concret dans un contexte SOC (SecureBank)

L'attaquant n'a pas besoin de connaître le mot de passe d'Alice. Grâce à l'injection $ne, MongoDB retourne son compte et l'API répond "success":true, "role":"analyst". Toutes les actions suivantes (consultation d'incidents, accès aux logs, déclenchement d'alertes) sont journalisées sous l'identité d'Alice. La traçabilité des incidents est corrompue — l'enquête forensique pointe vers un utilisateur innocent.

2.2 Injection via $regex et $gt — Exfiltration de données

📖 Opérateurs $regex et $gt

{"champ":{"$regex":"motif"}} : sélectionne les documents dont le champ correspond à l'expression régulière. Une regex vide "" correspond à toute chaîne.

{"champ":{"$gt":valeur}} : sélectionne les documents dont le champ est supérieur à la valeur. Pour les chaînes, $gt:"" sélectionne toute chaîne non vide.

// Injection $regex — retourne TOUS les logs (regex vide matche tout)
{ "src_ip": { $regex: "" } }

// Injection $gt — retourne tous les logs dont action > "" (= tous)
{ "action": { $gt: "" } }

// Injection ciblée — retourne les logs dont l'IP commence par "192"
{ "src_ip": { $regex: "^192" } }
⚠ Pourquoi c'est dangereux dans le contexte SOC SecureBank

La collection security_logs contient des informations opérationnelles hautement sensibles : adresses IP internes des serveurs bancaires, types d'actions SOC, statuts d'incidents, horodatages des interventions. Une exfiltration via $regex permet à un attaquant de :

  1. Cartographier le réseau interne de SecureBank (plages IP des serveurs)
  2. Identifier les failles actives (incidents au statut "investigating" depuis longtemps)
  3. Choisir ses prochaines cibles en connaissance de cause

2.3 Mass Assignment — Injection de champs non autorisés

📖 Définition — Mass Assignment

Le mass assignment se produit quand une application applique directement un objet utilisateur à une opération de mise à jour sans filtrer les champs modifiables. L'attaquant peut injecter des champs qu'il ne devrait pas contrôler : role, is_admin, account_balance…

// ❌ VULNÉRABLE — endpoint de mise à jour de profil
app.post('/users/update-profile', async (req, res) => {
  await db.collection('users').updateOne(
    { username: req.user.username },
    { $set: req.body }   // ❌ TOUT req.body est appliqué sans filtre !
  );
  res.json({ success: true });
});

// Utilisateur légitime envoie :   {"email":"nouveau@example.com"}
// Attaquant envoie :              {"email":"hacker@evil.com", "role":"admin"}
//
// MongoDB applique $set sur les DEUX champs :
//   → email est mis à jour (attendu)
//   → role passe à "admin"  (non attendu — élévation de privilège silencieuse !)
//
// L'attaquant vient de s'auto-promouvoir administrateur
// sans aucun message d'erreur, sans aucune alerte !
⚠ Erreur de conception fréquente

Confondre "l'utilisateur envoie ce qu'il veut modifier" avec "je dois modifier tout ce qu'il envoie". Ces deux affirmations sont fondamentalement différentes. La règle absolue : c'est l'application qui définit les champs modifiables, jamais l'utilisateur. L'utilisateur peut proposer des valeurs — il ne choisit pas les champs.

🔲 Checkpoint 2 — Démonstration contrôlée

Q1. Pourquoi {"password":{"$ne":""}} contourne-t-il le filtre de login ?

  • A) MongoDB accepte toujours les opérateurs $ne sans vérification de sécurité.
  • B) Express.js ne valide pas automatiquement les corps de requêtes JSON.
  • C) La condition "password ≠ ''" est vraie pour tout utilisateur ayant un mot de passe non vide, ce qui correspond à tous les comptes réels.
  • D) Le champ password n'existe pas dans la collection users de soc_lite.
✅ C est correct. L'opérateur $ne:"" demande "retourne les documents où password est différent d'une chaîne vide". Tout utilisateur ayant un mot de passe non vide satisfait cette condition — et c'est le cas de tous les comptes réels. La vérification du mot de passe est entièrement contournée.

Q2. Qu'est-ce qui rend le mass assignment particulièrement dangereux en contexte bancaire ?

  • A) Il rend MongoDB instable et peut provoquer des corruptions de données.
  • B) Il efface automatiquement les logs d'audit existants.
  • C) Il est détecté immédiatement par les systèmes de monitoring standards.
  • D) Un utilisateur peut s'auto-attribuer des permissions élevées (ex: role:admin) de façon silencieuse, sans aucune alerte.
✅ D est correct. Le mass assignment permet de modifier des champs sensibles comme role lors d'une opération de mise à jour apparemment légitime (ex: changement d'email). Cette élévation de privilège ne génère pas d'erreur, n'est pas détectée par les outils standards, et peut rester invisible jusqu'à ce que l'attaquant exploite ses nouveaux droits.

3Correction : Validation, Allow-list et Filtre Sûr

Trois défenses complémentaires et séquentielles à appliquer systématiquement avant toute requête MongoDB construite depuis une entrée utilisateur. Ces trois niveaux constituent un pipeline de validation : si l'une des étapes échoue, la requête est rejetée avec une erreur HTTP 400.

3.1 Validation des types

📖 Règle fondamentale

Si vous attendez une chaîne de caractères, vérifiez que c'est bien une chaîne — pas un objet, pas un tableau, pas un entier. Un objet {"$ne":""} n'est pas une chaîne et doit être rejeté immédiatement, avant tout traitement. C'est la première ligne de défense.

// Validation de type — à placer en tête de chaque endpoint
function isCleanString(val) {
  if (typeof val !== 'string') return false;  // rejette objects, arrays, numbers
  if (val.trim().length === 0) return false;    // rejette les chaînes vides
  if (val.length > 100) return false;           // protection contre les payloads longs
  return true;
}

// Validation de format IPv4 — pour le champ src_ip
function isValidIPv4(val) {
  if (typeof val !== 'string') return false;
  // Le pattern exige exactement 4 groupes de 1 à 3 chiffres séparés par des points
  return /^(\d{1,3}\.){3}\d{1,3}$/.test(val) &&
         val.split('.').every(n => +n <= 255);
}

// Exemples :
isCleanString("alice")          // true  ✔
isCleanString({ $ne: "" })      // false ✔ — rejeté (typeof objet, pas chaîne)
isValidIPv4("192.168.1.10")    // true  ✔
isValidIPv4({ $regex: "" })   // false ✔ — rejeté dès typeof

3.2 Filtrage des opérateurs MongoDB ($-filtering)

📖 Convention des opérateurs MongoDB

Par convention, tous les opérateurs MongoDB commencent par le caractère $ ($ne, $gt, $regex, $where…). Toute clé commençant par $ dans un objet reçu de l'utilisateur est une tentative d'injection — ou tout au moins, un input invalide. Il faut la détecter et refuser la requête.

// Vérification récursive — aucun opérateur MongoDB dans l'objet reçu
function hasNoMongoOperators(obj) {
  if (typeof obj !== 'object' || obj === null) return true;  // scalaires : ok
  for (const key of Object.keys(obj)) {
    if (key.startsWith('$')) return false;           // opérateur détecté → rejet
    if (!hasNoMongoOperators(obj[key])) return false;  // vérification récursive
  }
  return true;
}

// Tests de démonstration :
hasNoMongoOperators({ username: "alice" })           // true  ✔
hasNoMongoOperators({ password: { $ne: "" } })      // false ✔ (clé $ne détectée)
hasNoMongoOperators({ a: { b: { $gt: 5 } } })       // false ✔ (imbriqué, détecté récursivement)
hasNoMongoOperators("chaîne simple")               // true  ✔ (scalaire)

3.3 Le pattern safeFilter — code complet corrigé

📖 Principe du safeFilter

Ne jamais réutiliser l'objet utilisateur comme filtre MongoDB. Construire un nouvel objet en extrayant et validant chaque champ individuellement, un par un. Seuls les champs que l'application connaît et valide explicitement peuvent entrer dans le filtre. Tout champ non prévu est silencieusement ignoré.

const express = require('express');

const app = express();

// Allow-list des valeurs autorisées pour le champ status
const ALLOWED_STATUSES = ['open', 'closed', 'investigating'];

// ─── ENDPOINT 1 : /logs/search SÉCURISÉ ─────────────────
app.post('/logs/search', async (req, res) => {
  const safeFilter = {};  // ✅ objet construit par l'APPLICATION, pas par l'utilisateur

  // Champ src_ip : doit être une adresse IPv4 valide
  if (req.body.src_ip !== undefined) {
    if (!isValidIPv4(req.body.src_ip))
      return res.status(400).json({ error: 'src_ip invalide — format IPv4 requis' });
    safeFilter.src_ip = req.body.src_ip;   // ✅ string validée, copiée telle quelle
  }

  // Champ action : doit être une chaîne alphanumérique
  if (req.body.action !== undefined) {
    if (!isCleanString(req.body.action))
      return res.status(400).json({ error: 'action invalide' });
    safeFilter.action = req.body.action;   // ✅
  }

  // Champ status : doit être dans l'allow-list explicite
  if (req.body.status !== undefined) {
    if (!ALLOWED_STATUSES.includes(req.body.status))
      return res.status(400).json({ error: 'status non autorisé' });
    safeFilter.status = req.body.status;   // ✅
  }

  // Tout autre champ de req.body est silencieusement ignoré
  // → champ "role", "admin", "$where"... = inexistant pour le filtre

  try {
    const results = await db.collection('security_logs')
                            .find(safeFilter)       // ✅ filtre entièrement maîtrisé
                            .project({ _id:0, ts:1, src_ip:1, action:1, status:1 })
                            .limit(20)
                            .toArray();
    res.json({ count: results.length, data: results });
  } catch (err) {
    res.status(500).json({ error: 'Erreur serveur' }); // ✅ ne pas exposer err.message
  }
});

// ─── ENDPOINT 2 : /auth/login SÉCURISÉ ──────────────────
app.post('/auth/login', async (req, res) => {
  // ✅ Validation de type en PREMIER — avant toute opération
  if (typeof req.body.username !== 'string' || typeof req.body.password !== 'string')
    return res.status(400).json({ error: 'username et password doivent être des chaînes' });

  const username = req.body.username.trim();
  const password = req.body.password;

  if (!isCleanString(username) || password.length === 0)
    return res.status(400).json({ error: 'Entrée invalide' });

  try {
    // ✅ On cherche uniquement par username (string validée)
    // et on compare le mot de passe côté JavaScript — pas dans le filtre MongoDB
    const user = await db.collection('users').findOne(
      { username },   // filtre minimal : string validée
      { projection: { _id:0, username:1, password:1, role:1 } }
    );
    // ✅ Comparaison de chaînes en JS — ne peut PAS être manipulée par $ne
    if (!user || user.password !== password)
      return res.status(401).json({ success: false, message: 'Identifiants incorrects' });

    res.json({ success: true, message: `Bienvenue ${user.username}`, role: user.role });
  } catch (err) {
    res.status(500).json({ error: 'Erreur serveur' });
  }
});
✅ Les 3 clés du code corrigé
  1. Validation de type en premier : typeof req.body.password !== 'string' rejette tout objet JSON avant d'entrer dans la logique métier. Le filtre n'est jamais construit si l'entrée est invalide.
  2. safeFilter construit clé par clé : aucun champ imprévu ne peut entrer dans le filtre. L'application contrôle entièrement ce qu'elle envoie à MongoDB.
  3. Comparaison JavaScript du mot de passe : user.password !== password est une comparaison de chaînes en JavaScript — elle ne peut pas être manipulée par un opérateur MongoDB. Le mot de passe n'est plus dans le filtre MongoDB.

4Projection minimale et principe « Least Data »

📖 Principe du moindre privilège sur les données

Même si les injections sont bloquées, une API qui retourne tous les champs d'un document expose des données sensibles inutilement. La projection minimale consiste à déclarer explicitement les champs à inclure dans la réponse — et à n'inclure que le strict nécessaire pour la fonction appelante.

Retourner password_hash, _id, ou des notes internes dans une réponse JSON est une fuite de données, même si l'injection est bloquée. C'est une deuxième couche de défense indépendante.

// ❌ Pas de projection → retourne TOUS les champs du document
db.collection('users').findOne({ username: 'alice' })
// Retourne : { _id: ObjectId("..."), username: "alice",
//              password: "motdepasse123",    ← FUITE !
//              role: "analyst",
//              internal_notes: "ticket_12"   ← FUITE !
//            }

// ✅ Projection minimale → retourne seulement le nécessaire
db.collection('users').findOne(
  { username: 'alice' },
  { projection: { _id:0, username:1, role:1 } }
)
// Retourne : { username: "alice", role: "analyst" }
//            → password et notes internes : absents ✔

// Pour l'endpoint /logs/search — seuls 4 champs opérationnels nécessaires :
.find(safeFilter)
.project({ _id:0, ts:1, src_ip:1, action:1, status:1 })
.limit(20)
⚠ Le _id doit toujours être exclu (sauf besoin explicite)

Par défaut, MongoDB inclut le champ _id dans toutes les réponses. C'est un identifiant interne (ObjectId) qui n'a aucune valeur pour le client front-end — mais qui peut aider un attaquant à construire des requêtes MongoDB directes. L'exclure systématiquement : _id:0.

5Checklist Anti-NoSQLi et Résumé de séance

Cette séance a couvert le cycle complet d'une vulnérabilité NoSQL Injection : mécanisme technique, démonstration contrôlée, correction et prévention. Le hardening infrastructure (Séance 3) est nécessaire mais insuffisant — la sécurité applicative est une couche complémentaire obligatoire, indépendante et non substituable.

📋 Checklist de sécurité — API Node.js + MongoDB
  • ✅
    Valider le type de chaque entrée : typeof val === 'string' — rejeter systématiquement les objets et tableaux là où une chaîne est attendue. C'est la première et la plus importante des défenses.
  • ✅
    Ne jamais faire find(req.body). Construire un safeFilter en extrayant et validant chaque champ individuellement. L'application doit contrôler entièrement ce qu'elle envoie à MongoDB.
  • ✅
    Appliquer une allow-list de valeurs pour les champs à domaine fini (status, rôle, action) : Array.includes() ou enum explicite. Toute valeur hors liste retourne HTTP 400.
  • ✅
    Valider les formats par regex pour les champs contraints : adresses IPv4, dates ISO, UUIDs, emails. Rejeter immédiatement tout format non conforme.
  • ✅
    Filtrer les clés $ : toute clé commençant par $ dans un objet reçu d'un utilisateur est une tentative d'injection d'opérateur MongoDB. Refuser la requête.
  • ✅
    Projection minimale sur tous les find() et findOne() : exclure explicitement _id:0, password_hash, et tout champ non nécessaire à la fonction appelante.
  • ✅
    Ne jamais exposer err.message dans la réponse API — retourner un message générique côté client, logger le détail complet côté serveur (Winston, Morgan…). Les messages d'erreur MongoDB détaillés aident l'attaquant.
  • ✅
    Pour les mises à jour : construire l'objet $set avec des champs nommés explicitement — jamais $set:req.body. L'application définit les champs modifiables.
  • ❌
    À ne jamais faire : find(req.body), findOne(req.body), $set:req.body, res.json({error:err.message}). Ces pratiques sont des vulnérabilités connues.
  • ⚠️
    En production : utiliser bcrypt (ou Argon2) pour stocker et comparer les mots de passe. La comparaison en clair de ce TP est uniquement pédagogique — elle ne doit jamais figurer dans un code de production.
