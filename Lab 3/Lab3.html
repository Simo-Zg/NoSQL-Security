Partie 1 — Surface d'attaque MongoDB

1.1 Sécurité de la base de données vs sécurité de l'application

Définition — Sécurité de la base de données La sécurité de la base de données désigne l'ensemble des mécanismes qui protègent le système de gestion de base de données (SGBD) lui-même : authentification des clients, autorisation des opérations, chiffrement des données au repos et en transit, et contrôle de l'exposition réseau.

Il est essentiel de comprendre que la sécurité d'un système d'information repose sur plusieurs couches complémentaires. La sécurité applicative concerne le code de l'application (validation des entrées, gestion des sessions, protection contre les injections au niveau de l'API ou du front-end). La sécurité de la base de données, en revanche, concerne la couche de persistance elle-même : qui peut se connecter, quelles opérations sont autorisées, sur quelles données, et depuis quel réseau.

Ces deux couches ne se substituent pas l'une à l'autre. Une application parfaitement sécurisée ne protège pas contre un accès direct à la base de données si celle-ci est exposée sans authentification. Inversement, une base de données correctement durcie ne protège pas contre une faille dans le code applicatif (par exemple, une injection NoSQL exploitée via l'API de l'application).

Dans le contexte d'un SOC (Security Operations Center), les données stockées dans la base — journaux de sécurité, incidents, informations sur les utilisateurs — sont des actifs sensibles. Un accès non autorisé en lecture permettrait à un attaquant de comprendre les capacités de détection de l'organisation. Un accès en écriture permettrait de modifier ou supprimer des preuves d'intrusion. La protection de la base de données est donc un enjeu direct de l'intégrité du processus de réponse aux incidents.

Critère Sécurité applicative Sécurité de la base de données
Périmètre Code, API, interface utilisateur SGBD, fichiers de données, protocole réseau
Authentification Utilisateurs finaux (login/mot de passe, SSO) Comptes techniques et humains se connectant au SGBD
Autorisation Rôles métier dans l'application Rôles SGBD (RBAC) : droits sur collections/opérations

Responsable Développeur, DevSecOps DBA, administrateur système, DevSecOps
Piège fréquent — « L'application vérifie déjà les droits »

Une erreur courante consiste à considérer que, puisque l'application gère ses propres rôles (administrateur, analyste, lecteur), il n'est pas nécessaire de configurer des comptes séparés au niveau de MongoDB. Ce raisonnement est dangereux : si un attaquant obtient un accès réseau au port MongoDB (par exemple via un mouvement latéral), il contourne entièrement l'application et ses contrôles. Sans authentification ni RBAC au niveau de la base, toutes les données sont alors accessibles en lecture et en écriture.

Impact sécurité : compromission totale des données (confidentialité, intégrité, disponibilité) sans aucune trace dans les journaux applicatifs.

Question de compréhension

Q : Un pare-feu applicatif (WAF) protège-t-il contre un accès direct au port 27017 de MongoDB depuis le réseau interne ? Justifier.

Réponse attendue : Non. Un WAF (Web Application Firewall) filtre le trafic HTTP/HTTPS à destination de l'application web. Un accès direct au port 27017 utilise le protocole MongoDB Wire Protocol, qui n'est pas inspecté par un WAF. Seuls des contrôles au niveau réseau (pare-feu réseau, segmentation, binding) et au niveau du SGBD (authentification, RBAC) protègent contre ce vecteur.

1.2 Pourquoi « MongoDB sans auth » est dangereux, même en local

Définition — Mode sans authentification Par défaut, une instance MongoDB démarrée sans le paramètre --auth (ou sans configuration équivalente) accepte toute connexion entrante sans exiger d'identifiants. Tout client capable d'atteindre le port d'écoute obtient un accès complet (lecture, écriture, administration).

Ce comportement par défaut a une raison historique : simplifier la prise en main et le développement local. Cependant, il constitue un risque majeur, même dans un environnement qui semble isolé.

Risque en réseau local (LAN) : dans un réseau d'entreprise ou un réseau universitaire, de nombreuses machines partagent le même sous-réseau. Si une instance MongoDB écoute sur 0.0.0.0:27017 sans authentification, toute machine du même réseau peut s'y connecter. Un poste compromis, un stagiaire curieux, ou un script automatisé de reconnaissance interne suffisent pour accéder à l'intégralité des données. Les attaques par mouvement latéral exploitent précisément ce type de service exposé sans contrôle.

Risque sur le poste local : même sur 127.0.0.1, une application malveillante installée sur le poste (un navigateur compromis exécutant du JavaScript malicieux, un malware, ou un outil de développement piégé) peut initier une connexion vers localhost:27017. De plus, certaines configurations Docker publient par défaut le port sur 0.0.0.0 (toutes les interfaces), rendant la base accessible au-delà de la machine hôte sans que l'utilisateur en soit conscient.

Précédents documentés : entre 2016 et 2017, des dizaines de milliers d'instances MongoDB exposées sur Internet sans authentification ont été victimes de rançongiciels automatisés. Les attaquants, utilisant de simples scripts, se connectaient, copiaient les données, les supprimaient, puis déposaient une note de rançon. Ces attaques ne nécessitaient aucune compétence avancée : la simple absence d'authentification suffisait.

Principe de sécurité — Défense en profondeur

Ne jamais considérer qu'une seule couche de protection suffit. L'authentification, le RBAC, le contrôle réseau et la gestion des secrets sont des couches complémentaires. La compromission d'une couche ne doit pas entraîner la compromission totale du système. Ce principe, dit de défense en profondeur (defense in depth), est un fondement de la cybersécurité.

Exemple commenté — Connexion sans identifiants

Considérons une instance MongoDB lancée sans authentification :

# Lancement d'un conteneur MongoDB SANS authentification
docker run -d --name mongo_insecure -p 27017:27017 mongo:7

# Connexion depuis n'importe quel client réseau — aucun identifiant requis
mongosh "mongodb://localhost:27017"

# Une fois connecté, accès total :
use soc_lite
db.security_logs.find().limit(3)   // Lecture de tous les journaux
db.incidents.drop()                // Suppression de la collection entière
db.dropDatabase()                  // Destruction de la base complète

Ligne 2 : le paramètre -p 27017:27017 publie le port MongoDB sur toutes les interfaces de l'hôte (0.0.0.0). Toute machine pouvant atteindre l'hôte sur le port 27017 peut se connecter.
Ligne 5 : la chaîne de connexion ne contient ni nom d'utilisateur ni mot de passe. La connexion réussit immédiatement.
Lignes 8–10 : sans RBAC, le client dispose implicitement de droits d'administration totaux. Il peut lire les données sensibles, supprimer des collections entières, voire détruire la base. Dans un contexte SOC, cela signifie la perte des preuves d'investigation.

Piège fréquent — Docker et le binding par défaut

Lorsqu'on utilise docker run -p 27017:27017, Docker lie le port sur 0.0.0.0 par défaut (toutes les interfaces réseau). Beaucoup de développeurs pensent que le conteneur n'est accessible que localement. En réalité, si la machine possède une interface réseau connectée à un LAN ou à Internet, le port 27017 est exposé sur cette interface. Pour restreindre l'écoute à la boucle locale, il faut explicitement écrire -p 127.0.0.1:27017:27017.

Impact sécurité : exposition involontaire de la base à l'ensemble du réseau local, voire à Internet si la machine est directement connectée.

1.3 Figure 1 — Instance insecure vs instance secure

Interprétation : La figure 1 oppose deux configurations d'une même instance MongoDB hébergeant la base soc_lite. À gauche, la configuration insecure présente trois faiblesses cumulées : l'authentification est désactivée (auth: OFF), le binding est configuré sur toutes les interfaces (0.0.0.0), et le port 27017 est publié sans restriction. Tout client réseau — y compris un attaquant ayant compromis une machine du LAN — peut se connecter sans identifiant et obtenir un accès complet : lecture des journaux de sécurité, modification des incidents, voire suppression de la base entière via drop(). À droite, la configuration secure applique trois couches de protection complémentaires : l'authentification SCRAM est activée (seuls les comptes déclarés peuvent se connecter), le binding est restreint à 127.0.0.1 (seules les connexions locales sont acceptées), et le port n'est pas publié vers l'extérieur du conteneur Docker. L'utilisateur soc_analyst_ro, même après authentification, n'a accès qu'en lecture aux collections autorisées par son rôle. La collection users lui est invisible, et les opérations destructrices sont interdites. Cette comparaison illustre le principe de défense en profondeur : chaque couche réduit la surface d'attaque indépendamment des autres.

1.4 Activité 1 — Déployer une instance insecure puis une instance secure

Activité 1 — Observer la différence insecure vs secure

Objectif : constater concrètement qu'une instance MongoDB sans authentification accepte toute connexion, puis vérifier qu'une instance avec authentification la refuse.

Durée estimée : 25 minutes

Prérequis : Docker installé et fonctionnel, terminal disponible, aucune instance MongoDB en cours sur le port 27017.

Étape 1 — Démarrer une instance SANS authentification

# Nettoyage préventif (ignorer l'erreur si le conteneur n'existe pas)
docker rm -f mongo_insecure 2>/dev/null

# Lancement sans authentification, port publié sur toutes les interfaces
docker run -d --name mongo_insecure -p 27017:27017 mongo:7

Pourquoi : cette commande crée un conteneur MongoDB en mode par défaut (authentification désactivée). Le flag -p 27017:27017 publie le port sur 0.0.0.0.

Étape 2 — Se connecter sans identifiants et vérifier l'accès

# Connexion anonyme
mongosh "mongodb://localhost:27017"

# Dans le shell mongosh :
use soc_lite

# Insérer un document de test
db.security_logs.insertOne({
  timestamp: new Date(),
  source_ip: "10.0.0.99",
  event_type: "test_insecure",
  message: "Ce document a été inséré sans authentification"
})

# Vérifier que l'insertion a réussi
db.security_logs.find({ event_type: "test_insecure" })

# Vérifier les droits d'administration
db.adminCommand({ listDatabases: 1 })

Résultat attendu : la connexion réussit immédiatement. L'insertion fonctionne. La commande listDatabases retourne la liste de toutes les bases sans erreur. Cela démontre qu'un client anonyme dispose d'un accès administrateur complet.

🔴 Preuve à conserver : copier-coller la sortie de listDatabases dans votre rapport.

Étape 3 — Arrêter l'instance insecure

docker rm -f mongo_insecure

Étape 4 — Démarrer une instance AVEC authentification

# Lancement avec authentification activée et un admin initial
docker run -d --name mongo_secure \
  -p 127.0.0.1:27017:27017 \
  -e MONGO_INITDB_ROOT_USERNAME=adminSoc \
  -e MONGO_INITDB_ROOT_PASSWORD=Passw0rd_S3cure! \
  mongo:7

Pourquoi : les variables MONGO_INITDB_ROOT_USERNAME et MONGO_INITDB_ROOT_PASSWORD créent automatiquement un utilisateur administrateur dans la base admin et activent l'authentification. Le port est cette fois lié à 127.0.0.1 uniquement.

Étape 5 — Tenter une connexion anonyme (doit échouer)

# Tentative de connexion sans identifiants
mongosh "mongodb://localhost:27017"

# Dans mongosh, tenter une opération :
use soc_lite
db.security_logs.find()
# → Erreur attendue : requires authentication

Résultat attendu : la connexion TCP est acceptée (le protocole autorise la connexion initiale), mais toute opération sur les données échoue avec un message de type MongoServerError: command find requires authentication.

🔴 Preuve à conserver : copier-coller le message d'erreur dans votre rapport.

Étape 6 — Se connecter avec les identifiants corrects

# Connexion authentifiée
mongosh "mongodb://adminSoc:Passw0rd_S3cure!@localhost:27017/admin"

# Vérifier l'accès admin
db.adminCommand({ listDatabases: 1 })
# → Doit réussir cette fois

Résultat attendu : la connexion réussit. La commande listDatabases retourne la liste des bases. L'accès est fonctionnel uniquement avec les bons identifiants.

🔴 Preuve à conserver : copier-coller la sortie de listDatabases (avec connexion authentifiée).

1.5 Checkpoint 1

Checkpoint 1 — Vérification des acquis

Question 1 : Citer trois risques concrets liés à une instance MongoDB démarrée sans authentification sur un réseau local universitaire.

Réponse attendue : (1) Tout poste du réseau peut lire les données sensibles (ex. : journaux de sécurité, incidents). (2) Un attaquant (ou un utilisateur malveillant) peut modifier ou supprimer des données, compromettant l'intégrité des preuves. (3) Un script automatisé (ransomware, worm) peut découvrir le port 27017, supprimer les données et déposer une note de rançon — ce scénario a été observé massivement en 2016-2017.

Question 2 : Quelle est la différence entre -p 27017:27017 et -p 127.0.0.1:27017:27017 dans Docker ?

Réponse attendue : -p 27017:27017 publie le port sur toutes les interfaces réseau de l'hôte (0.0.0.0), le rendant accessible depuis le réseau local et potentiellement depuis Internet. -p 127.0.0.1:27017:27017 publie le port uniquement sur l'interface de boucle locale : seuls les processus de la machine hôte peuvent y accéder.

Question 3 : Pourquoi l'authentification au niveau de la base de données est-elle nécessaire même si l'application possède son propre système de login ?

Réponse attendue : Le système de login de l'application protège l'accès via l'interface applicative (HTTP). Cependant, si un attaquant obtient un accès réseau au port MongoDB (mouvement latéral, mauvaise configuration réseau), il contourne l'application et se connecte directement à la base. Sans authentification MongoDB, il obtient un accès complet. Les deux couches de contrôle sont complémentaires et nécessaires (principe de défense en profondeur).

Partie 2 — Authentification : admin initial et authDB

2.1 Mécanisme d'authentification MongoDB

Définition — Authentification L'authentification est le processus par lequel le serveur MongoDB vérifie l'identité d'un client qui tente de se connecter. Elle repose sur la présentation d'identifiants (nom d'utilisateur et mot de passe) associés à un compte déclaré dans une base de données d'authentification.

MongoDB stocke les comptes utilisateurs dans des bases de données spécifiques. Chaque utilisateur est défini dans une base de données, appelée sa base d'authentification (authentication database). Le couple (nom d'utilisateur, base d'authentification) forme l'identifiant unique d'un compte. Cela signifie qu'un utilisateur nommé alice dans la base admin est un compte distinct d'un utilisateur nommé alice dans la base soc_lite.

Lorsqu'un client se connecte, il doit indiquer trois informations : le nom d'utilisateur, le mot de passe, et la base d'authentification. Si l'une de ces trois informations est incorrecte, l'authentification échoue. C'est une source fréquente d'erreurs pour les débutants, qui oublient de spécifier la bonne base d'authentification.

Le mécanisme d'authentification par défaut de MongoDB est SCRAM (Salted Challenge Response Authentication Mechanism). D'autres mécanismes existent (x.509, LDAP, Kerberos), mais ils sortent du périmètre de cette séance d'introduction. SCRAM est le mécanisme adapté à un déploiement local ou de développement.

L'activation de l'authentification se fait de deux manières principales :

  • Via les variables d'environnement Docker (MONGO_INITDB_ROOT_USERNAME / MONGO_INITDB_ROOT_PASSWORD) : l'image officielle MongoDB détecte ces variables et crée automatiquement un utilisateur root dans la base admin, puis démarre le serveur avec l'authentification activée.
  • Via le fichier de configuration (mongod.conf) avec le paramètre security.authorization: enabled, combiné à la création manuelle de l'administrateur avant l'activation (ou en utilisant la localhost exception).

Dans le cadre de cette séance, la première méthode (variables d'environnement Docker) est privilégiée car elle est plus simple et moins sujette à erreur pour un débutant.

2.2 SCRAM — Principe conceptuel

Définition — SCRAM SCRAM (Salted Challenge Response Authentication Mechanism) est un protocole d'authentification par défi-réponse. Il permet au client de prouver qu'il connaît le mot de passe sans jamais transmettre le mot de passe en clair sur le réseau.

Le fonctionnement de SCRAM peut être résumé conceptuellement en quatre étapes (sans entrer dans les détails cryptographiques) :

  1. Stockage côté serveur : lors de la création du compte, le serveur ne stocke pas le mot de passe en clair. Il stocke une version dérivée du mot de passe, obtenue en appliquant une fonction de hachage itérée combinée à un sel (salt) aléatoire. Le sel garantit que deux utilisateurs ayant le même mot de passe auront des valeurs stockées différentes.
  2. Défi (challenge) : lors de la connexion, le serveur envoie au client un défi aléatoire unique (un nonce) ainsi que le sel associé au compte.
  3. Réponse (response) : le client calcule une preuve cryptographique à partir du mot de passe saisi par l'utilisateur, du sel reçu et du défi. Cette preuve est envoyée au serveur.
  4. Vérification : le serveur effectue le même calcul avec les informations qu'il possède. Si les résultats correspondent, l'authentification est acceptée. Le mot de passe n'a jamais transité sur le réseau.
Intérêt sécurité de SCRAM

SCRAM offre trois propriétés de sécurité importantes : (1) le mot de passe ne circule jamais en clair (protection contre l'interception réseau) ; (2) le serveur ne stocke pas le mot de passe en clair (protection en cas de compromission de la base de comptes) ; (3) chaque session utilise un défi aléatoire différent (protection contre les attaques par rejeu).

Cependant, SCRAM ne protège pas contre un mot de passe faible. Si un attaquant obtient les données stockées côté serveur (le hash salé), il peut tenter une attaque par dictionnaire ou force brute hors-ligne. Le choix d'un mot de passe robuste reste indispensable.

Piège fréquent — Confondre « chiffrement du protocole » et « authentification »

SCRAM assure que le mot de passe ne transite pas en clair, mais il ne chiffre pas le contenu des requêtes et des réponses. Les documents insérés ou lus circulent en clair sur le réseau. Pour chiffrer l'ensemble du trafic, il faut activer TLS/SSL (hors périmètre de cette séance). L'authentification et le chiffrement du transport sont deux mécanismes distincts et complémentaires.

Impact sécurité : sans TLS, un attaquant en position d'écoute réseau (ex. : ARP spoofing sur le LAN) peut lire le contenu des requêtes MongoDB même si l'authentification est activée.

2.3 Création de l'administrateur initial

L'administrateur initial est le premier compte créé sur une instance MongoDB sécurisée. Ce compte possède le rôle root dans la base admin, ce qui lui confère tous les privilèges sur toutes les bases. Son existence est une condition préalable à la création de comptes plus restreints.

L'ordre des opérations est critique. Il existe deux approches :

Approche 1 — Variables d'environnement Docker (recommandée pour les débutants)

# L'image officielle crée l'admin et active l'auth automatiquement
docker run -d --name mongo_secure \
  -e MONGO_INITDB_ROOT_USERNAME=adminSoc \
  -e MONGO_INITDB_ROOT_PASSWORD=Passw0rd_S3cure! \
  mongo:7

Ligne 3 : la variable MONGO_INITDB_ROOT_USERNAME définit le nom du compte administrateur.
Ligne 4 : la variable MONGO_INITDB_ROOT_PASSWORD définit son mot de passe.
Comportement automatique : l'image Docker crée un utilisateur avec le rôle root dans la base admin et démarre mongod avec l'authentification activée. L'étudiant n'a pas besoin de gérer la séquence manuellement.

Approche 2 — Création manuelle (pour comprendre le mécanisme)

Cette approche est présentée à titre pédagogique. Elle illustre la localhost exception : lorsque l'authentification est activée mais qu'aucun utilisateur n'existe encore, MongoDB autorise une connexion depuis localhost pour créer le premier administrateur. 

# 1. Démarrer MongoDB avec auth activée mais SANS utilisateur initial
docker run -d --name mongo_manual -p 127.0.0.1:27017:27017 mongo:7 \
  mongod --auth --bind_ip_all

# 2. Se connecter en local (localhost exception active)
mongosh "mongodb://localhost:27017"

# 3. Créer l'administrateur dans la base admin
use admin
db.createUser({
  user: "adminSoc",
  pwd: "Passw0rd_S3cure!",
  roles: [{ role: "root", db: "admin" }]
})

# 4. La localhost exception se désactive automatiquement
# 5. Se déconnecter et se reconnecter avec les identifiants
quit()
mongosh "mongodb://adminSoc:Passw0rd_S3cure!@localhost:27017/admin"
ARRÊT — Activer l'auth AVANT de créer l'admin

Si l'on active --auth et qu'on oublie de créer l'administrateur (et que la localhost exception a déjà été consommée ou que l'on n'est pas en local), on se retrouve face à une instance inaccessible. Aucune connexion ne sera autorisée. La seule solution est de redémarrer l'instance sans l'authentification, créer l'admin, puis réactiver l'authentification. En production, cela implique un arrêt de service.

Règle absolue : toujours s'assurer qu'un administrateur existe (ou que la localhost exception est utilisable) avant ou au moment de l'activation de l'authentification.

2.4 La notion d'authenticationDatabase

Définition — authenticationDatabase La authentication database (base d'authentification) est la base de données dans laquelle un compte utilisateur a été créé. Elle constitue une composante de l'identité de l'utilisateur : MongoDB identifie un compte par le couple (username, authenticationDatabase).

Lors de la connexion, le client doit indiquer la base d'authentification. Cette information peut être transmise de deux manières : 

# Méthode 1 : dans l'URI de connexion (après le dernier /)
mongosh "mongodb://adminSoc:Passw0rd_S3cure!@localhost:27017/admin"
#                                                        ^^^^^
#                                         authenticationDatabase = admin

# Méthode 2 : avec le paramètre --authenticationDatabase
mongosh "mongodb://localhost:27017" \
  -u adminSoc \
  -p "Passw0rd_S3cure!" \
  --authenticationDatabase admin

Si la base d'authentification n'est pas spécifiée, mongosh utilise par défaut la base indiquée dans l'URI (ou test si aucune n'est indiquée). Si le compte n'existe pas dans cette base, l'authentification échoue avec le message Authentication failed — même si le mot de passe est correct.

Piège fréquent — Oublier la base d'authentification

L'erreur la plus courante pour un débutant est de tenter la connexion suivante : 

# ❌ Erreur : authenticationDatabase implicite = "test" (par défaut)
mongosh "mongodb://adminSoc:Passw0rd_S3cure!@localhost:27017"

Le compte adminSoc a été créé dans la base admin. Sans spécifier /admin dans l'URI, le client tente de s'authentifier contre la base test, où ce compte n'existe pas. L'authentification échoue systématiquement. Le message d'erreur Authentication failed ne précise pas la cause exacte, ce qui rend le diagnostic difficile pour un débutant.

Impact sécurité : ce n'est pas une faille de sécurité en soi, mais cette confusion conduit fréquemment les développeurs frustrés à désactiver l'authentification « pour que ça marche », ce qui crée une véritable faille.

Partie 3 — RBAC : moindre privilège et séparation des comptes

3.1 Principe du moindre privilège

Définition — Principe du moindre privilège (Least Privilege) Le principe du moindre privilège stipule que chaque utilisateur, processus ou programme ne doit disposer que des droits strictement nécessaires à l'accomplissement de sa tâche, et aucun droit suppl��mentaire. Tout privilège excédentaire augmente la surface d'attaque.

Ce principe, formalisé par Jerome Saltzer en 1974, est l'un des fondements de la sécurité informatique. Appliqué aux bases de données, il implique que chaque compte technique ou humain ne doit pouvoir accéder qu'aux collections et aux opérations dont il a besoin. Un analyste SOC qui consulte des journaux n'a pas besoin de pouvoir les modifier. Une application qui écrit des logs n'a pas besoin de pouvoir administrer le serveur.

Conséquences pratiques : au lieu de créer un unique compte « fourre-tout » avec le rôle root utilisé par tous les acteurs (application, analystes, managers, scripts), on crée des comptes séparés avec des rôles précis. Si un compte est compromis (mot de passe divulgué, session volée), les dégâts sont limités au périmètre de ses droits.

Analogie concrète : dans un bâtiment sécurisé, chaque employé reçoit un badge donnant accès uniquement aux zones nécessaires à son travail. Le technicien de maintenance accède aux locaux techniques mais pas à la salle des coffres. Le caissier accède à la salle des coffres mais pas aux serveurs informatiques. Si un badge est volé, seules les zones autorisées par ce badge sont compromises, et non l'ensemble du bâtiment.

MongoDB implémente ce principe via un système de RBAC (Role-Based Access Control — contrôle d'accès basé sur les rôles). Chaque utilisateur se voit attribuer un ou plusieurs rôles, et chaque rôle définit un ensemble de privilèges (combinaison d'une ressource et d'une action).

Impact sécurité — Compte unique vs comptes séparés

Considérons un scénario dans lequel un seul compte root est utilisé par l'application web, les scripts d'analyse et les managers. Si les identifiants de ce compte fuitent (par exemple, codés en dur dans un fichier docker-compose.yml versionnné sur un dépôt public), l'attaquant obtient un accès administrateur total : lecture de toutes les données, suppression, modification de la configuration, création de nouveaux comptes, voire exécution de commandes système dans certaines configurations. Avec des comptes séparés, la fuite du mot de passe du compte soc_analyst_ro ne compromet que la lecture de certaines collections — les données restent intègres et le serveur n'est pas administrable.

3.2 Rôles built-in et rôles personnalisés

Définition — Rôle MongoDB Un rôle est un ensemble nommé de privilèges. Un privilège associe une ressource (base de données, collection, ou cluster) à une ou plusieurs actions (find, insert, update, remove, createCollection, dropDatabase, etc.). Les rôles peuvent être built-in (prédéfinis par MongoDB) ou user-defined (personnalisés).

MongoDB fournit un ensemble de rôles prédéfinis couvrant les cas d'usage courants. Les plus importants pour cette séance sont présentés dans le tableau suivant :

Rôle built-in Périmètre Actions principales autorisées Cas d'usage
read Une base de données find, listCollections, listIndexes Consultation en lecture seule (analyste, reporting)
readWrite Une base de données find, insert, update, remove, createCollection, createIndex Application qui lit et écrit des données
dbAdmin Une base de données Gestion des index, statistiques, validation, compactage Administration de la structure d'une base (sans lire les données)
userAdmin Une base de données Création et gestion des utilisateurs et rôles Gestion des comptes sur une base spécifique
readAnyDatabase Toutes les bases find sur toutes les bases Monitoring, audit global
root Cluster entier Toutes les actions sur toutes les ressources Administration complète (à utiliser avec parcimonie)

Les rôles built-in sont pratiques mais parfois trop larges. Par exemple, le rôle readWrite sur soc_lite donne accès en lecture/écriture à toutes les collections de cette base. Si un compte ne doit écrire que dans incidents et pas dans security_logs, un rôle built-in ne suffit pas. Il faut alors créer un rôle personnalisé (user-defined role).

Création d'un rôle personnalisé — Syntaxe générale

# Connexion en tant qu'admin
     mongosh "mongodb://adminSoc:Passw0rd_S3cure!@localhost:27017/admin"
     
     # Se positionner sur la base où le rôle sera défini
     use soc_lite
     
     # Créer un rôle personnalisé
     db.createRole({
       role: "incidentManager",          // Nom du rôle
       privileges: [
         {
           resource: { db: "soc_lite", collection: "incidents" },
           actions: ["find", "insert", "update", "remove"]
         }
         // Aucun accès aux autres collections
       ],
       roles: []                          // Pas d'héritage d'autres rôles
     })

Ligne 8 : la commande db.createRole() est exécutée dans le contexte de la base soc_lite. Le rôle est donc défini dans cette base.
Lignes 10–14 : le tableau privileges définit précisément les ressources et actions autorisées. Ici, seules les opérations find, insert, update et remove sont autorisées, et uniquement sur la collection incidents.
Ligne 16 : le champ roles: [] signifie que ce rôle ne hérite d'aucun autre rôle. On peut aussi composer des rôles en incluant des rôles existants dans ce tableau.

Piège fréquent — Confondre « base du rôle » et « base de la ressource »

Un rôle est défini dans une base de données (celle dans laquelle on exécute db.createRole()). Mais les privilèges qu'il accorde peuvent porter sur n'importe quelle base et collection, via le champ resource. Il ne faut pas confondre la base de définition du rôle avec la base sur laquelle il accorde des droits. En pratique, il est recommandé de définir les rôles personnalisés dans la même base que les données qu'ils protègent, pour plus de lisibilité.

Impact sécurité : une confusion sur le périmètre d'un rôle peut conduire à accorder des droits sur la mauvaise base ou la mauvaise collection, violant le principe du moindre privilège.

3.3 Politique RBAC pour la base SOC-lite

La base soc_lite contient trois collections avec des niveaux de sensibilité différents :

  • security_logs : journaux de sécurité bruts. Écrits par l'application de collecte, consultés par les analystes. Intégrité critique (les logs ne doivent pas être modifiés après écriture).
  • incidents : fiches d'incidents créées et mises à jour par les analystes et les managers. Confidentialité et intégrité importantes.
  • users : informations sur les utilisateurs du SI surveillé. Données personnelles, accès restreint.

La politique RBAC suivante respecte le principe du moindre privilège en définissant quatre profils distincts :

Compte Profil métier security_logs incidents users Admin serveur
adminSoc Administrateur BD R/W R/W R/W ✅ Oui (root)
soc_app Application de collecte R/W R/W ❌ Aucun ❌ Non
soc_analyst_ro Analyste SOC (lecture) Read Read ❌ Aucun ❌ Non
soc_manager Responsable incidents ❌ Aucun R/W ❌ Aucun ❌ Non

Justification de chaque profil :

  • soc_app est le compte utilisé par l'application de collecte. Elle doit écrire les journaux entrants dans security_logs et créer/mettre à jour des incidents dans incidents. Elle n'a aucun besoin d'accéder aux données utilisateurs ni d'administrer le serveur.
  • soc_analyst_ro est le compte d'un analyste SOC de niveau 1. Il consulte les journaux et les incidents pour ses investigations, mais ne doit pas pouvoir les modifier (principe d'intégrité des preuves). Il n'accède pas à la collection users.
  • soc_manager est le compte du responsable qui gère le cycle de vie des incidents (création, mise à jour du statut, clôture). Il n'a pas besoin d'accéder aux logs bruts (les analystes les synthétisent dans les incidents) ni aux données utilisateurs.
Pourquoi la collection users n'est accessible à aucun compte applicatif ?

La collection users contient des informations personnelles sur les utilisateurs du système d'information surveillé (noms, adresses email, identifiants). Conformément aux principes de minimisation des données (RGPD, principe du moindre privilège), seul l'administrateur doit y accéder directement. Si l'application ou les analystes ont besoin de résoudre un identifiant utilisateur, un mécanisme applicatif dédié (API, vue, agrégation contrôlée) doit être mis en place plutôt qu'un accès direct à la collection.

3.4 Figure 2 — Diagramme RBAC : utilisateurs → rôles → collections

Interprétation : La figure 2 illustre la politique RBAC conçue pour la base soc_lite. Chaque colonne représente un niveau d'abstraction : à gauche les comptes utilisateurs, au centre les rôles qui leur sont assignés, puis les collections sur lesquelles ces rôles octroient des droits, et enfin les actions autorisées. Le compte soc_app se voit attribuer un rôle readWrite limité aux collections security_logs et incidents, lui permettant les quatre opérations CRUD. Le compte soc_analyst_ro dispose du rôle read sur ces mêmes collections, restreint à la seule opération find. Le compte soc_manager utilise un rôle personnalisé (incidentManager) qui n'accorde de droits que sur la collection incidents. La collection users, représentée en rouge, n'est accessible par aucun compte applicatif — seul l'administrateur adminSoc (trait pointillé, rôle root) y a accès. Cette séparation stricte garantit que la compromission d'un compte ne permet pas d'accéder à l'ensemble des données ni d'administrer le serveur.

     # → C'est une limitation du rôle built-in (voir remarque ci-dessous)
     
     # ❌ TEST REFUSÉ : tentative d'administration
     db.adminCommand({ listDatabases: 1 })
     # → MongoServerError: not authorized
Observation importante — Limitation des rôles built-in

Les rôles built-in read et readWrite s'appliquent à toutes les collections d'une base de données. Ainsi, soc_app avec readWrite sur soc_lite peut aussi lire et écrire dans users, ce qui n'est pas souhaitable. Pour corriger cela, il faudrait remplacer le rôle built-in par un rôle personnalisé, comme cela a été fait pour soc_manager. Cette amélioration est proposée dans le devoir maison.

Impact sécurité : un rôle trop large viole le principe du moindre privilège. En environnement de production, il est recommandé de toujours utiliser des rôles personnalisés délimitant précisément les collections accessibles.

3.6 Checkpoint 3

Checkpoint 3 — Vérification des acquis

Question 1 : Expliquer la différence entre un rôle built-in et un rôle personnalisé. Donner un cas concret où un rôle built-in est insuffisant.

Réponse attendue : Un rôle built-in est prédéfini par MongoDB et couvre des cas d'usage génériques (ex. : read = lecture sur toute une base). Un rôle personnalisé est défini par l'administrateur et permet un contrôle fin, collection par collection, action par action. Cas concret : le rôle readWrite sur soc_lite donne accès à toutes les collections, y compris users. Si l'on veut limiter l'écriture à security_logs et incidents seulement, un rôle personnalisé est nécessaire.

Question 2 : Le compte soc_analyst_ro est compromis (mot de passe divulgué). Quelles données sont exposées ? Quelles données restent protégées ?

Réponse attendue : Avec le rôle read sur soc_lite, l'attaquant peut lire toutes les collections de la base (logs, incidents, et malheureusement users avec le rôle built-in). En revanche, il ne peut pas modifier ni supprimer de données (l'intégrité est préservée), il ne peut pas accéder aux autres bases, et il ne peut pas administrer le serveur. Avec un rôle personnalisé plus strict (excluant users), les données personnelles seraient également protégées.

Question 3 : Pourquoi est-il déconseillé d'utiliser le compte adminSoc (root) pour l'application de collecte de logs en production ?

Réponse attendue : Le compte root dispose de tous les privilèges sur toutes les bases et le cluster. Si les identifiants de ce compte fuitent (code source, logs, fichier de configuration), l'attaquant obtient un contrôle total : lecture/modification/suppression de toutes les données, création de comptes, modification de la configuration serveur. En utilisant un compte dédié (soc_app) avec des droits limités, la portée d'une compromission est réduite aux seules opérations de collecte.
